NAME

npsync - synchronize files and directories  

SYNOPSIS

npsync [OPTION]... SRC... DST  

DESCRIPTION

Synchronize SRC(s) and DST on storages with non-POSIX interfaces.

SRC(s) and DST should exist (see description of storage type `npsync://' for exception). DST is parent for SRC copy. Use option -h to print available storage interfaces.

Interface utilities used to access storage(s) are run with timeout. SIGTERM is sent in 10 minutes, SIGKILL is sent in 15 minutes of utility execution wall-clock time.  

OPTIONS

-a
Archive mode, the same as `-r -c -d mtime,size'.
--aux-npsync FILE
Use source item attributes and checksums recorded in auxiliary catalogue FILE instead of storage. Storage attributes and checksums are used if absent in catalogue.
-c
The same as `--create-dirs --create-files'.
--copied FILE
Add to FILE names of source files and directories (to be) copied to DST.
--create-dirs
Create absent destination directories existing in source.
--create-files
Create absent destination files existing in source.
-d CRITERIA
Set difference criteria for source-destination file comparison. CRITERIA is a comma-separated list of words (without spaces): mtime, size, cks.
* mtime
if destination modification time older.
* size
is file sizes are equal. If mtime has been tested, then size is compared only if mtimes are equal.
* cks
is file checksums are equal.
Default list is empty, so no files are compared.
--exclude PATTERN
Exclude source files matching extended bash(1) PATTERN.
--exclude-file FILE
Exclude source items read from FILE.
--exclude-regex PATTERN
Exclude source files matching regex(3) PATTERN.
-f
Force execution, ignore errors.
--failed-cp FILE
Add to FILE names of source files failed to be copied to destination.
--failed-mkdir FILE
Add to FILE names of source directories failed to be created in destination.
--failed-sync-attrs FILE
Add to FILE names of source items failed to synchronize attributes in destination.
-i
Prompt before every modification.
-h
Print help and exit.
--include PATTERN
Include source files matching extended bash(1) PATTERN.
--include-file FILE
Include source items read from FILE.
--include-regex PATTERN
Include source files matching regex(3) PATTERN.
-l LEVEL
Copy nested directories with LEVEL as the maximum level of nesting (by default the number is zero i.e. unlimited).
--npsync-interface FILE
Load npsync storage interface from FILE.
--omit-copy-cks-check
Omit check of copied file checksums.
-p FILE
Add source file and directory attributes to FILE. The FILE can be used later with prefix `npsync://' as source or destination.
NB: The option turns on checksum calculation for all source files.
NB2: The option can't be used simultaneously with `-t'.
-r
Synchronize and/or copy nested directories recursively.
--real-data-cks
Calculate checksum of real data when possible instead of getting stored value
-s ATTRS
The same as
`--sync-dir-attrs ATTRS --sync-file-attrs ATTRS'.
--sync-dir-attrs ATTRS
Synchronize directory attributes in destination with source. ATTRS is a comma-separated list of words (without spaces): mode, mtime, owner.
* mode
file mode bits.
* mtime
modification time.
* owner
uid and gid.
--sync-file-attrs ATTRS
Synchronize file attributes in destination with source. ATTRS is a comma-separated list of words (without spaces): mode, mtime, owner.
* mode
file mode bits.
* mtime
modification time.
* owner
uid and gid.
--synced FILE
Add to FILE names of source files and directories with attributes (to be) synchronized in destination. List generated in test mode (files and dirs with attrs to be synced) may differ since:
*
Items absent in destination can't be checked (when option `-c' is given).
*
File copying changes mtime of parent dir.
-t
Test mode, don't perform real synchronization and/or copying, simply report potential problems and discover what is to be synchronized and/or copied.
NB: The option can't be used simultaneously with `-p'.
-v
Verbose output. Multiple -v options increase the verbosity. Default level is equal zero. The maximum is equal 3.
--version
Print version and exit.
 

INCLUDE/EXCLUDE PATTERNS AND FILES

Paths used in pattern matching are source paths. Files exclude-file and include-file if pointed are to contain paths separated by newline character. Each pattern option can be assigned only once. Exclude-file and include-file can be assigned some times, contents of the files are added.

There are three policies to match file paths:

1.
Neither patterns nor exclude-file nor include-file are assigned. Include all files by default.
2.
An exclude pattern or file assigned first in command line. Matching algo:
2.1
Include all files by default.
2.2
If path to file matches an exclude pattern (bash or regex) or is read from exclude-file exclude it then check if file matches an include pattern (bash or regex) or is read from include-file include it.
3.
An include pattern or file assigned first in command line. Matching algo:
3.1
Exclude all files by default.
3.2
If path to file matches an include pattern (bash or regex) or is read from include-file include it then check if file matches an exclude pattern (bash or regex) or is read from exclude-file exclude it.

Filtering of directory paths works the same way with exclude-file and include-file only. Pattern matching for directory paths is not performed.

CONSEQUENCES OF ERRORS

If npsync is prematurely terminated by a signal or error, files or file hierarchies may be only partially copied and/or files and directories may have incorrect permissions, modification time, owner.  

APPLICATION USAGE

Npsync implements a small subset of rsync features, so if SRC and DST are placed on storages with POSIX interfaces then use of rsync can be smarter. Data copying is verified always with checksum matching like rsync does. The utility was designed to operate with multi-terabyte nested directories placed on storages of different types so only regular files and directories are supported. No extended attributes are synchronized. Use an archiver to keep not regular files and extended attributes or small files efficiently.

Destination file or directory is ignored if it is absent in source. An error happens in inverse case. If SRC is alone swap it with DST in argument list and run npsync in test mode to discover destination files or directories absent in source, e.g.:

npsync -a -t -f a/b/item c/d
npsync -a -t -f c/d/item a/b

Be careful, attribute synchronization can change destination directory mtime so a destination file absent in source can become newer than its parent directory.

Catalogue can be used as SRC or DST. The only difference is file copying, catalogue doesn't keep files, so can't be used as a source for file copying. Directory creation is provided by catalogue as a source with options `-r --create-dirs -f', but mtime may differ from original because attributes of parent directories can be synchronized before child creation. So mtime is to be synced after directory creation with options `-r -s mtime -f'.

Some storages provide checksum as a file attribute (e.g. CASTOR HSM). If the checksum is provided then it is used by default. To force checksum calculation for real data option `--real-data-cks' is to be used (e.g. to check real data integrity on storage medium). Use the option not frequently to prolong storage durability.

The version supports checksum type adler32 only.

HSM system can provide a utility to pre-stage files to disk from tape, to reorder tape operations in an optimal way. CASTOR HSM provides utility `stager_get' to reduce overall file access time for files migrated to tapes. At first list of accessed files is to be created with option `--copied'. SRC is an absolute CASTOR path in the example below:

npsync -a -t --copied FILELIST SRC DST

Accessed files are to be pre-staged:

stager_get -f FILELIST -U mytag

Pre-staging is performed asynchronously. Utility `stager_qry' is to be used to query status of pre-staged files:

stager_qry -U mytag

At last pre-staged files are accessed:

npsync -a SRC DST

Option `--aux-npsync' can be used for synchronization to get source item attributes and checksums from catalogue instead of storage. Number of IO operations is reduced significantly in the case.

Option --omit-copy-cks-check' is meant for advanced optimizations. It turns off source checksum calculation and copy checksum comparison and should not be used in common case. Correct optimization is to compare checksums for an extra stage after data transfer on destination side locally with catalogue. It may help to reduce load on transfer channel between source and destination twice and more.

Options `--failed-cp', `--failed-mkdir', `--failed-sync-attrs' are to be used to automatize error handling.  

EXAMPLES

1.
Attribute comparison. Destination attributes are compared with source. Case to discover what destination attributes are to be synchronized:
npsync -r -t -f -s mode,mtime,owner SRC DST
2.
Common comparison. Report potential problems and discover what source files and directories are to be copied. Case before mirroring to test enviromnent and access to source(s). Case to discover what files and directories were added or updated in source and absent or outdated in destination:
npsync -a -t -f SRC DST
3.
Full comparison. Checksums of each pair source -- destination are calculated and compared. Case to discover silent data corruptions:
npsync -r -t -f -d mtime,size,cks SRC DST
4.
Attribute synchronization. Destination attributes are synchronized with source. Case to restore destination attributes if they were lost (e.g. CASTOR doesn't keep seconds in mtime so seconds can be restored after copying from CASTOR):
npsync -r -s mode,mtime,owner -v SRC DST &> DST.out
NB:
Be careful, outdated destination after attribute synchronization can't be updated with common mirroring. Only full mirroring or comparison can discover mismatched files.
5.
Common mirroring. Case of outdated destination:
npsync -a -v -v SRC DST &> DST.out
NB:
Catalogue as a source provides comparison with destination attributes only. No destination files can be replaced.
6.
Full mirroring. Checksums of each pair source -- destination are calculated and compared. Destination file is replaced with source if checksums differ. No files and directories absent in destination are copied from source(s), only error messages about the items are printed. Case to restore from source(s) (e.g. from back-up) after storage errors (e.g. silent data corruptions) in destination:
npsync -r -c -d mtime,size,cks -v -v SRC DST &> DST.out
NB:
Catalogue as a source provides comparison with destination attributes only. No destination files can be replaced
7.
Interactive selection of mirrored files and directories. Case to speed up interactive selection with two stage execution. First stage is to create interactively list of mirrored files and directories:
npsync -a -t -i --copied FILE SRC DST
Second one is to mirror with the list:
npsync -a --include-file FILE SRC DST
8.
Catalogue creation. Catalogue can be used as a source or destination. The only difference is file copying, catalogue doesn't keep files, so can't be used as a source for file copying. Case to create catalogue:
npsync -a SRC npsync://SRC.npsync
9.
Catalogue clearing. File used with -p option is not recreated so can contain multiple outdated records for the same file or directory being used some times with modified source. If file and directory modification history doesn't matter the records can be removed. Case to create copy of catalogue without outdated records:
npsync npsync://orig.npsync npsync://copy.npsync
NB:
Order of records in catalogue can change after its modification.
10.
Full offline comparison. Catalogue can be used for comparison like file storage as a source and destination without limitations. Moreover comparison in the case is performed much faster. It is not required a lot of I/O operations to get attributes and checksums from catalogue. Case to perform full comparison by catalogues only:
npsync -d mtime,size,cks -s mode,mtime,owner -t npsync://first.npsync npsync://second.npsync
NB:
Equivalence can be tested yet faster with catalogue checksum comparison. Old records in the case are to be removed from catalogues as described above and checksums are to be calculated and compared for sorted catalogues:
sort -n catalogue|md5sum
 

BUGS

Npsync doesn't accept names with newline char '\n'. The char is used by design as Internal Field Separator (IFS) for directory listing.  

FUTURE DIRECTIONS

List of storage interfaces implemented currently includes FS with POSIX interface, CASTOR HSM system, NPSYNC catalogue. The interfaces are implemented with virtual functions to storage utilities, so npsync was designed to extend the list easily for any type of storage with tree-like representation of contents by interface utilities. A list of possible cases includes:
*
Storage without POSIX interface with interface utilities (e.g. a remote, cloud storage).
*
FS with unusable or absent kernel driver. Interface utilities could provide access to storage without the driver.
*
FS implemented with Linux FUSE. Interface utilities could bypass FUSE to accelerate access to the FS.
*
File and directory names can be converted somehow for syncing with storage function _item_path. So a storage interface can provide specific path composition or translate names from one language to another.
*
The utility was written in bash, so can be easily adapted by user e.g. to perform additional actions for synchronization or copying.
